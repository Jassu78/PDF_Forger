Section 01

Executive Summary

PDFForge is a fully offline, open-source Android PDF toolkit designed as a production-grade system capable of handling millions of users with zero cloud dependency. Every PDF operation — creation, compression, conversion, merging, splitting — runs entirely on-device using a hybrid JVM + NDK architecture.

500+Pages Supported
Android 10+Minimum SDK 29
Apache 2.0License
⚙️

Hybrid Engine Strategy

MuPDF (C/NDK) for core PDF rendering and manipulation. Apache PDFBox (JVM) for structural PDF operations. Apache POI for DOCX/PPTX conversion. Tesseract NDK for optional OCR.

🏗️

Clean + Modular Architecture

Strict Clean Architecture with layered modules: :feature, :domain, :data, :engine, :common. Plugin-style tool system enables community feature contributions without touching core.

🔒

Privacy by Design

No files stored internally without explicit user opt-in. All output via Android Storage Access Framework. Scoped storage compliance. Automatic temp file cleanup. No analytics that require network.

📦

APK Size Strategy

ABI splits reduce base from ~180MB to ~60MB per architecture. Dynamic Feature Modules for OCR (+20MB) and Conversion (+18MB) loaded on-demand. R8 full mode for dead code elimination.

🧵

Memory Safe Processing

Page streaming instead of full document load. BitmapRegionDecoder for large images. WorkManager for background operations with progress tracking. Strict coroutine scope management prevents leaks.

⚠️

Honest Offline Limitations

PDF to DOCX conversion quality is heuristic-based, not perfect. Complex multi-column layouts may misalign. Scanned PDF text extraction requires Tesseract (accuracy ~85-95%). No offline equivalent to cloud AI layout analysis.

Clean Architecture Flow

The dependency rule is enforced: outer layers depend on inner layers, never the reverse. Domain layer has zero Android framework imports — it's pure Kotlin.

Domain Layer Rules

  • No Android imports (no Context, no Bitmap)
  • Repository interfaces only (no impl)
  • UseCase per operation, single responsibility
  • Returns sealed Result<T> types
  • Coroutine suspend funs only
  • Fully unit testable without device

Engine Layer Rules

  • JNI wrappers are thin — business logic in Kotlin
  • All native resources wrapped in Closeable
  • Page streaming, never full doc in memory
  • Each engine isolated in its own module
  • Interface-based — engines are swappable
  • Error handling maps native codes to domain errors

Plugin Tool System

Each PDF tool is registered as a PdfTool plugin. Adding a new tool requires only: implement interface, register in DI, add screen. No changes to core modules.

// :domain:core — Tool plugin contract interface PdfTool { val id: String val nameRes: Int val iconRes: Int val category: ToolCategory suspend fun execute(params: ToolParams): OperationResult fun validate(params: ToolParams): ValidationResult fun estimateOutputSize(params: ToolParams): Long } // Example: New tool added in 10 lines class RotatePagesTool @Inject constructor( private val engine: MuPdfEngine ) : PdfTool { override val id = "rotate_pages" override suspend fun execute(params: ToolParams) = engine.rotatePages(params as RotateParams) }

Critical: MuPDF License Strategy

⚠️
MuPDF is AGPL-3.0. This means the entire app's source must be AGPL-3.0 compliant if using MuPDF's free license. This actually aligns with an open-source app — you must release your source code, which you intend to do anyway. However, if monetizing via a closed app, a commercial Artifex license is required (~$5K/year). Alternative: Use iText7 Community (AGPL) or custom-compile pdfium-android (BSD-3) for rendering while using PDFBox for manipulation. pdfium-android is BSD but lacks write capabilities.

Alternative Engine Analysis

EngineLicenseCapabilitiesVerdict
MuPDF AGPL / Commercial Full read/write, forms, annotations, compression, excellent large PDF handling RECOMMENDED
Apache PDFBox Apache 2.0 Full read/write, metadata, fonts, no NDK but slower on large docs COMPLEMENTARY
pdfium-android BSD-3 Read-only rendering from Chrome's PDF engine. No write support. VIEWER ONLY
iText7 Community AGPL Good PDF creation/editing, Java-only, larger than PDFBox ALTERNATIVE
LibreOffice Mobile MPL-2.0 Conversion, ~40MB binary, complex integration AVOID — too heavy

Threading Model

// Coroutine dispatcher strategy per operation type object AppDispatchers { // PDF reading/writing — heavy I/O val PDF_IO = Dispatchers.IO.limitedParallelism(2) // Image processing — CPU bound, limit to avoid OOM val IMAGE_PROC = Dispatchers.Default.limitedParallelism(2) // OCR — extremely CPU heavy, single thread val OCR = Dispatchers.Default.limitedParallelism(1) // SAF write operations val SAF_IO = Dispatchers.IO } // WorkManager worker with coroutine support class CompressPdfWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) { override suspend fun doWork(): Result { return try { val inputUri = getInputUri() val result = withContext(AppDispatchers.PDF_IO) { compressor.compress(inputUri, params) } setProgressAsync(workDataOf("progress" to 100)) Result.success(workDataOf("output_uri" to result.uri)) } catch (e: OutOfMemoryError) { // Retry with lower memory profile Result.retry() } catch (e: Exception) { Result.failure(workDataOf("error" to e.message)) } } }

ANR Prevention

UI Thread Rules (Strict)

  • Zero file I/O on main thread
  • No bitmap decoding on main thread
  • StrictMode enabled in debug builds
  • All use cases are suspend functions
  • Progress updates via StateFlow to UI
  • Background work via WorkManager only

Battery Optimization

  • WorkManager uses BATTERY_NOT_LOW constraint for batch ops
  • CPU governor hint via performanceHint API (Android 12+)
  • Limit parallel workers: max 2 simultaneous operations
  • Release resources immediately after page processing
  • No background polling or wake locks
  • Cancellation via CancellationSignal propagated to NDK

Bitmap Handling Strategy

// Safe large image loading — prevents OOM fun safeDecodeBitmap(uri: Uri, targetWidth: Int): Bitmap? { val opts = BitmapFactory.Options().apply { inJustDecodeBounds = true } contentResolver.openInputStream(uri).use { BitmapFactory.decodeStream(it, null, opts) } opts.inSampleSize = calculateInSampleSize(opts, targetWidth) opts.inJustDecodeBounds = false opts.inPreferredConfig = Bitmap.Config.RGB_565 // saves 50% vs ARGB_8888 for photos return contentResolver.openInputStream(uri).use { BitmapFactory.decodeStream(it, null, opts) } } // For images too large for BitmapFactory, use BitmapRegionDecoder fun tileDecodeAndProcess(uri: Uri, processor: (Bitmap) -> Unit) { val decoder = BitmapRegionDecoder.newInstance( contentResolver.openInputStream(uri)!!, false ) val tileHeight = 512 for (y in 0 until decoder.height step tileHeight) { val region = Rect(0, y, decoder.width, minOf(y + tileHeight, decoder.height)) val tile = decoder.decodeRegion(region, null) processor(tile) tile.recycle() // explicit recycle — critical } decoder.recycle() }

Threat Model & Mitigations

ThreatMitigationSeverity
Malicious PDF exploit (buffer overflow via crafted PDF) MuPDF's fz_try/fz_catch isolates all parsing. Process isolation in separate :engine process. ASLR enabled by default on Android. Validate magic bytes before any parsing. HIGH
Corrupted PDF causing crash fz_try catches all MuPDF exceptions. OOM handler in WorkManager retries with lower memory. Crash caught before reaching UI. MEDIUM
Temp file data leakage All temp files written to getCacheDir() (app-private). Cleanup registered in finally block. WorkManager completion handler deletes all temps. No external temp writes. MEDIUM
User file access beyond selection SAF URIs used exclusively. No REQUEST_INSTALL_PACKAGES, no MANAGE_EXTERNAL_STORAGE. App cannot read files user didn't explicitly pick. LOW — by design
Network data leakage INTERNET permission not declared in manifest. App cannot make network calls. NDK code has no network APIs. Verified via network security config. NONE — no network perm
Font/image bomb (zip bomb in PDF) Max extracted size limit: 500MB total per operation. fz_context memory limit enforced. Process killed if exceeds threshold. MEDIUM

Development Roadmap

Phase 1 — Core (Months 1-4)

Image to PDF, Merge, Split, Extract Pages, Rotate Pages, Basic Compression (quality reduction + metadata strip). MuPDF integration. SAF storage. Hilt DI. Compose UI with tool cards. GitHub Actions CI.

Phase 2 — Conversion (Months 4-7)

PDF to DOCX (Apache POI), DOCX to PDF (PDFBox), PPTX to PDF. Advanced compression (font subsetting, object streams). Dynamic Feature Module for conversion. Progress tracking UI. WorkManager for large files.

Phase 3 — OCR + Advanced (Months 7-10)

Tesseract OCR integration as DFM. Scanned PDF text extraction. Multi-language OCR support. PDF viewer with annotation support. Page reorder drag-and-drop UI.

Phase 4 — Forms + Signing (Months 10-14)

PDF form filling (AcroForms). Digital signature support (self-signed certificates). Password protection. Accessibility improvements. Android tablet UI. Performance optimization pass.

Section 11

Risk Analysis & Tradeoffs

🔴 HIGH: MuPDF AGPL License Compliance

If contributors or downstream users misunderstand AGPL requirements, forks could create compliance issues. Mitigation: Clear CONTRIBUTING.md, automated license checks in CI via FOSSA. Alternative path: migrate to full PDFBox stack (slower but clean Apache 2.0).

🔴 HIGH: PDF-to-DOCX Conversion Quality

Offline layout analysis is heuristic. Complex multi-column PDFs, PDFs with custom fonts, or heavily formatted documents will produce imperfect DOCX output. No offline solution matches cloud AI-based layout analysis. Mitigation: Clear UI warning, "best effort" labeling, provide raw text extraction fallback.

🟠 MEDIUM: APK Size on Non-Play Distributions

Fat APK (all ABIs) would be ~180MB. This matters for F-Droid or direct APK distribution. Mitigation: Build separate per-ABI APKs in release script. F-Droid can serve ABI-split APKs.

🟠 MEDIUM: MuPDF Native Crash Propagation

Corrupted PDFs or memory exhaustion can cause native SIGSEGV that bypasses Java exception handling. Mitigation: Process isolation (separate :engine process), fz_try/fz_catch around all MuPDF calls, crash reporting via local log only (no network).

🟠 MEDIUM: Apache POI Memory Usage on Large DOCX

POI loads entire document into memory. A 500-page DOCX can use 300-500MB RAM during conversion. Mitigation: SXSSF streaming API for large outputs, strict 200MB POI heap limit, reject inputs over 100MB with clear error.

🟢 LOW: Tesseract Accuracy on Non-Latin Scripts

Tesseract accuracy for Hindi, Arabic, CJK scripts is lower (~75-85%) compared to Latin (~92-96%). Mitigation: Per-language accuracy disclosure in UI, allow users to select language pack, quality threshold setting.

🟢 LOW: Dynamic Feature Module Delivery Failure

On-demand DFM download requires network connection. Mitigation: Clear "requires internet for first-time download" notice, graceful offline detection, allow manual APK with all DFMs included.

Section 12

Final Recommendation Stack

🔤

Language

Kotlin 1.9+ for all Kotlin code. C/C++17 via NDK r26 for engine layer. No Java in new code. Kotlin DSL for all build files.

⚙️

PDF Engine

MuPDF (AGPL) for all core PDF manipulation via JNI. Apache PDFBox 3.x for structural operations where MuPDF is overkill. Hybrid approach gives best performance + coverage.

🔄

Conversion

Apache POI 5.x for DOCX/PPTX output. PDFBox for DOCX→PDF and PPTX→PDF. Tesseract 5.x NDK for OCR (DFM). No LibreOffice — too heavy.

🧵

Async

WorkManager 2.9+ for all background operations with lifecycle-safe progress. Kotlin Coroutines with limited parallelism dispatchers. Never block main thread.

🏗️

Architecture

Clean Architecture + Modular Gradle. Strict layer isolation. Hilt for DI. Plugin-style PdfTool interface for extensibility. Convention plugins for build consistency.

🖥️

UI

Jetpack Compose 1.5+. Material Design 3. Dark mode first. Navigation Compose with type-safe routes. Compose drag-reorder for page ordering.

💾

Storage

Android SAF exclusively for file I/O. No MANAGE_EXTERNAL_STORAGE. DataStore for preferences. TempFileManager for scoped cleanup. No implicit file caching.

📦

Distribution

AAB for Play Store with ABI splits. Per-ABI APKs for F-Droid and GitHub Releases. Dynamic Feature Modules for OCR and Conversion. Target base APK: <75MB.

📋

License

AGPL-3.0 (matches MuPDF). All dependencies compatible. FOSSA license scan in CI. If pure Apache 2.0 needed: replace MuPDF with PDFBox-only (accept 2-3× performance reduction on large PDFs).

✅
Bottom line: This architecture is production-ready. MuPDF + PDFBox hybrid gives best-in-class offline PDF operations. The Clean + Modular architecture enables community contributions per module. APK stays under 75MB base with DFMs for heavy features. The only genuine limitation is conversion quality — heuristic layout analysis will never match cloud ML, and users must be clearly informed of this.
